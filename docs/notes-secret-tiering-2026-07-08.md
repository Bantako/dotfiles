3 層目が本丸で、特に restic/B2 の資格情報は次の質問と直結します。

バックアップだけでいいか — いいえ、1 つ穴があります

バックアップが最後の砦として機能する条件は「エージェントのミスがバックアップ自体に届かない」ことです。現状 NAS の backup コンテナに restic パスワードと B2 キーがあり、hermes が NAS を操作できるなら、暴走や騙された操作が restic forget --prune やバケット削除まで届き得ます。そうなるとバックアップは防壁ではなく単なる同一障害域です。対策は明確で:

1. B2 のアプリケーションキーを deleteFiles 権限なしで発行し直す（B2 はケイパビリティ単位でキーを作れます）。これでエージェント側から見てバックアップが append-only になり、「ミスっても昨日には戻れる」が構造的に保証されます。今回の対策の中で費用対効果が最大です
 2. 破壊的動詞に承認を挟む — down -v、ボリューム削除、restic forget、DB drop 系。approval.py にパッチを当てる文化が既にあるので、その延長で deny/承認パターンを足すだけです。safe-tmp-deletes と同じ「即死ではなくゴミ箱」思想を API 操作にも広げるイメージです
 3. リストア訓練を一度やる — 復元したことのないバックアップは存在確認ができていないのと同じです。immich の pg_dump を別ディレクトリに実際に戻してみる、を一度やっておくと「バックアップだけでいい?」への答えに確信が持てます
 4. RPO の確認 — 夜間バックアップなので最悪 24 時間分は消えます。生活データ用途ならたぶん許容範囲ですが、「hermes が日中に書き込んだメモが消える」ケースだけは ~/.hermes の git 自動コミットが埋めてくれるので、実は既にカバーされています
